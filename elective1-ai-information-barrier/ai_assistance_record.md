# AI 辅助整理记录

> 本任务全程使用 Claude Code（DeepSeek V4 Pro 后端）辅助信息检索、整理和分析。以下如实记录关键 AI 交互过程。

---

## 关键 Prompt 1：多角度信息检索

### 我让 AI 帮我做了什么

```
联网搜索 AI 编程工具与 Agent 2026 年的最新进展，覆盖以下角度：
- 整体格局（Claude Code / Cursor / Copilot / Codex 排名与差异）
- SWE-bench 基准测试最新榜单
- 开源编码 Agent（OpenHands / Aider / Cline）的进展
- 国产方案（通义灵码 / DeepSeek / Trae）的现状
- 计费模式变化与成本分析
- MCP / AGENTS.md 标准化进展
- 比赛与活动机会
```

### AI 的回答是否可靠

AI 的 WebSearch 结果本身是可靠的——它返回的是真实网页的标题、链接和摘要。但需要注意：(1) 搜索结果的质量取决于搜索词和搜索引擎的覆盖范围，不是所有信息都能搜到；(2) AI 整理的综述文本中，数字和排名需要回到原始链接核实。

### 哪些内容我自己进行了核查

- SWE-bench 分数：AI 同时返回了"88.6%"（厂商口径）和"76.8%"（官方统一 harness 口径），这个矛盾引起了我的警觉。我通过对比两个来源的原文，发现差异来自评测设置的不同。这个过程被记录在了 [opportunity_judgement.md](opportunity_judgement.md) 的交叉验证实例中。
- 国产模型横评排名：AI 整理了一份排名表，但我注意到这只是雷峰网单家媒体的评测，任务样本小，于是将它的价值等级从"强信号"降为"中等信号"。
- Copilot 计费转型日期：AI 给出的"2026-06-01"我回到 GitHub 官方博客原文验证，确认一致。

### 哪些内容我没有完全采纳

- AI 整理综述时频繁使用"领先""最强"等主观表述，我全部替换为可量化的数据（具体分数、百分比、价格）。例如 AI 说"Claude Code 在深度编程方面独树一帜"，我改为"统一 harness 重测中头部模型集中在 70-77%，第 1 与第 8 仅差 6 分"。
- AI 对厂商宣传材料缺乏批判性——它会复述"效率提升 3 倍"之类的宣称而不加质疑。我补充了 DX 研究（7.76% 中位增益）作为对照，并把厂商宣称归入"噪音"层级。
- AI 对国内生态的英文媒体报道有 1-2 个月的时滞，我额外补充了中文一手信源。

### 我如何避免只复制 AI 总结

**四步法**：(1) 搜索拿到原始链接 → (2) 回到原始链接核对关键数字 → (3) 不同来源交叉对比（同一基准的两个分数、多个评测的排名）→ (4) 形成自己的判断并标注判断依据。这条工作流被总结在了 [reflection.md](reflection.md) 中。

## 关键 Prompt 2：对照官方评分标准逐条补漏

### 我让 AI 帮我做了什么

在初稿完成后，我让 AI 逐条对照 DEEP 营的官方评分标准，检查每个部分是否覆盖了所有必答问题。例如"主题选择"需要回答 5 个问题，"信息来源记录"每条需要 6 个字段。AI 帮我识别出了格式上的缺口（如 source_records 缺"为什么选择"字段、information_analysis 缺 9 个维度的显式标注）。

### AI 的回答是否可靠

格式检查本身是可靠的——AI 可以精确比对"要求列表"和"现有内容"。但格式对齐不等于内容质量提升。有些 AI 建议的修改是机械填空式的，需要人工判断是否真的增强了内容。

### 哪些内容我自己进行了核查

- 每个"补漏"后我重新读了改动的段落，确认新增内容不是为填而填。例如补"为什么选择这个来源"时，我给每条来源写的理由都基于该来源的实际特征（如"唯一一篇对 AGENTS.md 做大规模实证的论文"），而不是套模板。
- 灰色渠道警告部分的 6 种行为分类，我对照了 DeepSeek 和 OpenAI 的 Terms of Service 做了确认，确保"合规"的判断依据是真实的政策条款而非 AI 的猜测。

### 哪些内容我没有完全采纳

- AI 建议在主题选择中加入"我准备花多少时间"的具体承诺，但我认为在初次整理阶段做精确时间估算不现实且不诚实。我改为在行动建议中承诺具体动作而非时间量。
- AI 建议把每条信息来源都做成独立卡片格式（每个一张大表），但 15 条全做会导致文件过长。我保留了分组结构，只在每条内部补全 6 字段——在完整性和可读性之间取平衡。

### 我如何避免只复制 AI 总结

**对照法**：把 AI 产出的格式化内容和我自己最初的理解做对照。如果 AI 加了一条"我的判断"但我自己想不到，就追问自己：为什么没想到？是信息不够，还是没往那个方向想？然后补上自己的思考过程，而不是直接复制 AI 的判断。例如 AI 在价值判断中写了"国产模型已进入可用区间"这个结论，我追问了"可用区间的标准是什么"，然后补充了"与一线差距缩小到工程可接受范围 + 价格是零头"这个具体量化标准。

## 关键 Prompt 3：动手实验的设计与执行

### 我让 AI 帮我做了什么

```
写一个含两个 bug 的 JS 购物车函数，并调用 DeepSeek API 修复，
记录完整的耗时、token 消耗和成本数据，作为"低价 API 可行性"的一手验证。
```

### AI 的回答是否可靠

AI 生成的实验代码和 API 调用过程是我的直接操作——API 确实被调用了、bug 确实被修复了、测试确实通过了。这部分不存在"AI 回答是否准确"的问题，因为结果是可复现的。完整过程文件保留在 experiment/ 目录。

### 哪些内容我自己进行了核查

- 修复前后的代码差异：我逐行对比了 `buggy.js` 和 `fixed.js`，确认 AI 修复了 `i <= cart.length`（→ `i < cart.length`）和 `total > 100`（→ `total >= 100`）两个 bug
- Token 用量数据直接来自 API 返回的 `usage` 字段，未经 AI 转述
- 成本估算使用了雷峰网报道的 DeepSeek 定价区间（≈¥2/百万 Token），同时标注了这个数字来自二手报道、精确价格应以官方定价页为准

### 哪些内容我没有完全采纳

- AI 最初建议用 `content[0].text` 提取修复代码，但因为 deepseek-v4-pro 返回的是 `thinking + text` 双块结构，首次运行报错。我改为 `content.find(c => c.type === "text")` 后成功——这个改动证明了：AI 给的代码示例不一定适配实际模型行为，需要根据真实响应结构调整。这个坑被记录在了 [experiment_report.md](experiment_report.md) 的"附带发现"中。

### 我如何避免只复制 AI 总结

**亲手跑**：实验是我在本机实际执行的，不是 AI 描述的。API 调用失败、编码错误、响应结构解析失败——这些意外都是真实发生的，没有一个是在 AI 总结中能提前预料到的。这正是"动手验证"区别于"读 AI 总结"的核心价值。

## 总结：AI 辅助的边界

| AI 做了什么 | 我做了什么 |
|---|---|
| 搜索 80+ 网页、返回摘要和链接 | 筛选 15+4 条，逐条标注一手/二手属性 |
| 整理信息为结构化初稿 | 核查关键数字（回到原始链接对比）、发现口径冲突 |
| 对照评分标准识别格式缺口 | 判断每个补漏是否增强内容，拒绝机械填空 |
| 生成实验代码框架 | 实际执行、处理报错、分析真实数据 |
| 提出分析结论 | 追问"为什么"、补充量化标准、标注置信度 |

**一句话**：AI 是搜索引擎 + 格式化工具 + 对照检查清单，但筛选、核查、判断和动手执行全部由我完成。信息雷达的"雷达"部分是人的判断，AI 只是天线。
