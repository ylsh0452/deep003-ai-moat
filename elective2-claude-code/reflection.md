# Claude Code 使用反思

## 1. Claude Code 最适合帮我解决什么问题

经过这次任务，我认为 Claude Code 最适合解决的是"我知道要什么但自己做不出来"的问题——具体来说：
- **需求明确但实现我不会**：比如"做个仪表盘，有统计卡片、折线图、饼图、表单"。这些需求我能说清楚，但我不会写 Chart.js 配置，也不会写 localStorage 读写逻辑。Claude Code 填补了"能描述"和"能实现"之间的鸿沟。
- **重复性工程操作**：比如 git 提交、目录创建、文件写入——这些不需要创造力但容易出错的机械操作，Claude Code 做得比我快且准确。
- **从零到可运行的最小版本**：一次性生成的项目文件结构完整、可以直接打开浏览器验证，不用自己先搭脚手架。

**不适合的场景**：需要深度领域知识的任务（比如写学术论文中的方法论创新）、需要审美判断的任务（配色和排版需要人工微调）、需要真实外部验证的任务（基准测试数据必须回到原始来源核实）。

## 2. 它和普通 ChatGPT 对话有什么不同

| | 普通 ChatGPT 对话 | Claude Code |
|---|---|---|
| 产出形式 | 一段文字回答 | 完整的可运行文件（写入文件系统） |
| 验证方式 | 靠读文字判断对错 | 打开浏览器/跑命令直接验证 |
| 出错后 | 重新描述问题，从头生成 | 在原文件上修改、git 可追溯每次改动 |
| 最终成果 | 聊天记录（不可交付） | 可推送到 GitHub 的项目（可交付） |
| 上下文持续性 | 多轮对话容易丢失上下文 | 同一个项目内的上下文保持连贯 |

这次任务最直观的差异：如果我用普通 ChatGPT 做仪表盘，产出会是"复制这段代码到 index.html，然后……"——我需要自己创建文件、粘贴代码、调整路径、安装依赖。而 Claude Code 直接把这些都做了，我只需要打开浏览器验收。

## 3. 我在使用中遇到的最大困难是什么

不是代码，不是工具，是**把"模糊感觉"翻译成"精确需求"**。

最大的困难发生在一开始：我脑子里有一个"想追踪 Token 消耗"的想法，但不知道该怎么描述它才能让 AI 生成正确的代码。第一次描述是松散的——"做个好看的仪表盘，能看趋势和成本"。结果生成的东西不好用。

后来我把需求重新组织成了结构化 Prompt（我要做什么 / 使用场景 / 目标用户 / 需要包含的内容 / 不需要做什么），效果立刻好了。这个翻译过程——从大脑中的图景到穷举的功能清单——是最累的，但它也是最有价值的技能，因为它是唯一不能由 AI 代劳的环节。

第二个困难是**验收不彻底**。第一次生成后我草草看了一眼就截图了，没注意到成本列全部显示 ¥0.00。后来仔细看才发现是小数位不够。这件事教会我：AI 不会主动告诉你"这个显示可能有问题"——它完成了你要求的任务，但验证完全是你的事。

## 4. 我是否理解了它生成的所有内容？如果没有，哪些地方还不理解

**理解了的部分**：
- HTML 结构和 CSS 布局——这些很直观，看代码能对应到页面上的元素
- localStorage 读写逻辑——简单，就是 `getItem` 和 `setItem`
- `log.js` 的全部逻辑——命令行参数解析 + 文件追加写入，Node.js 基础知识

**不完全理解的部分**：
- Chart.js 的配置项。我知道 `type: 'line'` 是折线图、`type: 'doughnut'` 是环形图，但具体到 `tension: 0.3`（曲线平滑度）、`interaction.mode: 'index'`（悬停交互模式）这些参数，我只是知道"它让图表看起来对了"，但不理解为什么要用这个值。如果以后想修改图表样式，我需要去读 Chart.js 文档，而不能靠 AI 每次都生成对的东西。
- `--virtual-time-budget` 参数——我只知道它是让 Edge 等 JS 执行完再截图，不知道它和 `--timeout` 的区别。

**我的态度**：不完全理解是可以接受的。重要的是知道哪些地方不理解，以及去哪里找答案。我不需要变成 Chart.js 专家来使用这个仪表盘——我只需要知道"将来如果要修改图表，先去读 Chart.js 文档的对应章节"。

## 5. 如果别人也能使用 Claude Code，我和他们的差距在哪里

工具是同一个，差距在于三件事：

**需求描述的质量**。同样想做仪表盘，一个人说"帮我做个好看的 AI 成本页面"，另一个人说"4 个统计卡 + 折线图（X 轴日期 Y 轴 Token 数）+ 饼图（按模型分组）+ localStorage + 表单 + 深色主题 + 单文件"。后者的输出质量远高于前者。这种能力不是技术能力，是**把一个模糊问题分解成精确子需求**的能力。这次任务让我意识到，这个能力是可以通过练习提升的——每写一次结构化 Prompt，就是在做一次需求分解训练。

**验收的严格程度**。Claude Code 生成代码之后，一个人打开页面看一眼"嗯还行"就提交了，另一个人逐项检查：卡片数字对吗、图表渲染了吗、表单新增后数据在吗、刷新后数据丢了吗。两个人用的是同一个 AI，但后者的最终质量远高于前者。验收能力 = 知道什么是对的标准 + 愿意花时间逐项对照——这件事 AI 不会替你做的。

**持续改进的意愿**。一次生成就满足的人，得到的是一个一次性页面。发现成本显示 ¥0.00 不够直观、继续要求 AI 改为 4 位小数的人，得到的是一个不断逼近"满意状态"的成果。差距不在于 AI 能力，在于**你在"够用"和"满意"之间愿意投入多少轮迭代**。

## 6. 以后我准备如何继续提高 Claude Code 使用能力

三件事：

**第一，建自己的 Prompt 模板库**。这次任务中我发现了结构化 Prompt 的价值（我要做什么 / 使用场景 / 目标用户 / 需要包含的内容 / 希望的输出 / 不需要做什么）。以后每次给 Claude Code 派任务前，先花 3 分钟填这个模板。填不出来的部分说明需求还不够清晰——那就不急着开始，先把需求想清楚。

**第二，养成"验收清单"习惯**。每次 Claude Code 生成代码后，不直接打开就截图，而是先在心里列一张清单：这个页面应该有哪些功能？每个功能的预期行为是什么？然后逐项走一遍。清单内容不需要正式，记在脑子里就行——关键是"先想清楚什么是合格"再去看。

**第三，做越来越多的真实项目**。这次做仪表盘，下次做 MCP Server，再做任务网页——每做一个项目就积累一类场景的经验。做的项目类型越多，对"哪些任务适合 AI、哪些不适合"的判断就越准。目标是：半年后，不需要每次都在脑子里挣扎"这个需求该怎么描述才好"，大多数场景已经有直觉了。
